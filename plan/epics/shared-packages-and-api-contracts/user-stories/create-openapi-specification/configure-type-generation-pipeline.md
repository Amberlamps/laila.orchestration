# Configure Type Generation Pipeline

## Description

Set up the automated pipeline for generating TypeScript types from the OpenAPI 3.1 specification. This pipeline uses `openapi-typescript` to generate type definitions and configures `openapi-fetch` as the type-safe HTTP client that consumes these types.

The generated types ensure that API route handlers and client-side API calls are fully type-checked against the OpenAPI specification — any drift between the spec and implementation is caught at compile time.

## Acceptance Criteria

- [ ] `openapi-typescript` is installed as a devDependency in `packages/api-spec`
- [ ] `openapi-fetch` is installed as a dependency in `packages/api-spec` (used by consumers)
- [ ] `packages/api-spec/scripts/generate.ts` (or `generate.mjs`) script exists that runs `openapi-typescript` against `openapi.yaml`
- [ ] Running the generate script produces `packages/api-spec/generated/api.ts` with TypeScript type definitions
- [ ] Generated types include: `paths` (all endpoint paths with methods), `components` (all schema types), `operations` (all operation types)
- [ ] `packages/api-spec/package.json` includes a `generate` script: `"generate": "openapi-typescript openapi.yaml -o generated/api.ts"`
- [ ] `packages/api-spec/src/client.ts` exports a pre-configured `openapi-fetch` client factory:
  ```typescript
  // Creates a type-safe API client from the generated OpenAPI types
  import createClient from 'openapi-fetch';
  import type { paths } from '../generated/api';
  export function createApiClient(baseUrl: string) {
    return createClient<paths>({ baseUrl });
  }
  ```
- [ ] `packages/api-spec/src/index.ts` re-exports generated types and client factory
- [ ] Generated types are committed to the repository (not in `.gitignore`)
- [ ] `pnpm --filter @laila/api-spec generate` runs without errors and produces correct output

## Technical Notes

- `openapi-typescript` v7+ generates types compatible with `openapi-fetch` v0.10+
- The generated `paths` type is the primary interface consumed by `openapi-fetch` — it maps URL paths to request/response types
- Generation command: `npx openapi-typescript openapi.yaml -o generated/api.ts`
- The generated file should have a header comment indicating it is auto-generated:
  ```typescript
  // This file is auto-generated by openapi-typescript. Do not edit manually.
  // Regenerate with: pnpm --filter @laila/api-spec generate
  ```
- `openapi-fetch` provides a `createClient<paths>()` function that returns a fetch wrapper with `.GET()`, `.POST()`, `.PATCH()`, `.DELETE()` methods, each fully typed based on the path
- Example consumer usage:
  ```typescript
  // Type-safe API call — path, params, body, and response are all inferred
  const client = createApiClient('/api/v1');
  const { data, error } = await client.GET('/projects/{projectId}', {
    params: { path: { projectId: '...' } },
  });
  // data is typed as ProjectResponse, error is typed as ErrorResponse
  ```
- Consider adding a `prebuild` script that runs generation before build to ensure types are fresh
- The generation should be idempotent — running it twice should produce identical output
